## 回答重点

**索引不是越多越好**。索引是把双刃剑，读的时候能加速查询，写的时候却要付出代价。索引太多，时间和空间成本都会上去。

**从时间上看**，每次 INSERT、UPDATE、DELETE 都要同步更新索引。比如删一条 name 是"面试鸭"的记录，主键索引要改，name 字段如果有二级索引也要改，有 5 个索引就要改 5 棵 B+ 树。B+ 树还可能触发页分裂、页合并，写放大问题很严重。一张高并发的订单表如果建了 10 个索引，写入性能直接打骨折。

还有个容易忽略的点是查询优化器的负担。MySQL 执行 SQL 之前要分析走哪个索引成本最低，索引越多可选项越多，优化器耗时越长，甚至可能因为统计信息不准选错索引。

**从空间上看**，每个二级索引都是一棵独立的 B+ 树，每个数据页 16KB。假设一张表 1000 万行数据，一个二级索引差不多要占几百 MB 到几个 GB 的空间。索引一多，磁盘占用和内存开销都会飙升，Buffer Pool 里能缓存的热数据也变少了。

## 扩展知识

### 索引太多带来的具体问题

1）写性能急剧下降

InnoDB 用的是 B+ 树结构，插入数据时要维护树的有序性。如果插入的主键不是递增的，或者二级索引列的值分布随机，就会频繁触发页分裂。页分裂意味着要申请新的数据页、拷贝数据、更新指针，一个写操作可能变成好几个磁盘 I/O。索引越多，这种开销成倍放大。

![image.png](https://pic.code-nav.cn/mianshiya/question_picture/1783388929455529986/tgrR97TU_image_mianshiya.webp)

实测数据：同一张表，建 1 个索引和建 10 个索引，批量插入 100 万条数据的耗时可能差 3-5 倍。

2）优化器选择困难

MySQL 的查询优化器是基于成本的，它会估算每个可能的执行计划需要扫描多少行、多少次 I/O。索引越多，可选的执行计划越多，优化器要评估的组合就越多。复杂查询时，光是生成执行计划就要耗掉几十毫秒。

更麻烦的是统计信息不准的问题。MySQL 靠采样来估算索引的选择性，数据分布不均匀时估算容易出错，索引越多出错的概率越大，选错索引的查询可能比全表扫描还慢。

3）空间占用膨胀

二级索引存的是索引列的值加上主键值。如果主键是个 36 字节的 UUID，每个二级索引的每条记录都要额外存这 36 字节。一张表建 10 个索引，光是冗余存储的主键就要占用巨量空间。

4）DDL 操作变慢

加字段、改字段类型这些 DDL 操作，MySQL 要重建表。索引越多，重建的 B+ 树越多，DDL 耗时越长。一张几十 GB 的表如果有七八个索引，加个字段可能要跑好几个小时。

### 那索引该建多少个

没有标准答案，但有几个原则：

1）按需建索引，不要预防性建索引。先跑业务，等慢查询出来了再针对性建索引。

2）优先考虑联合索引。一个联合索引能覆盖多个查询场景，比建多个单列索引划算。联合索引 `idx_a_b_c` 可以支持 `WHERE a = ?`、`WHERE a = ? AND b = ?`、`WHERE a = ? AND b = ? AND c = ?` 三种查询。

3）单表索引数量一般控制在 5-6 个以内。阿里巴巴的 Java 开发手册建议单表索引不超过 5 个，联合索引字段不超过 5 个。

4）对于写多读少的表要更克制。日志表、消息表这种写入量大的场景，索引越少越好，甚至只留主键。

5）定期清理无用索引。可以用 `sys.schema_unused_indexes` 查看哪些索引长期没被用过，该删就删。

### 怎么判断索引是否该建

问自己几个问题：

这个字段的查询频率高不高？偶尔查一次的字段不值得建索引。

这个字段的选择性好不好？性别、状态这种只有几个值的字段，索引基本没用。

这个字段的写入频率高不高？频繁更新的字段建索引要谨慎。

现有索引能不能覆盖？已有联合索引的前缀字段不需要单独建索引。

## 面试官追问

#### 提问：索引选择性是什么？怎么判断一个字段适不适合建索引？

> 详细解答可以看：[什么样的字段适合加索引？索引失效了解吗？](https://www.mianshiya.com/question/1776483567317028866)

回答：索引选择性就是不重复的索引值数量除以表的总行数，越接近 1 说明选择性越好。可以用 `SELECT COUNT(DISTINCT column) / COUNT(*) FROM table` 来算。选择性低于 0.1 的字段基本不值得建索引，比如性别字段，全表就男女两个值，索引过滤效果约等于没有。

#### 提问：联合索引的字段顺序怎么确定？

> 详细解答可以看：[MySQL 中的覆盖索引和联合索引是什么？索引的最左前缀匹配原则是什么？](https://www.mianshiya.com/question/1772582920637227009)

回答：遵循三个原则。第一，把等值查询的字段放前面，范围查询的字段放后面，因为范围查询之后的字段用不上索引。第二，把选择性高的字段放前面，过滤效果更好。第三，考虑排序需求，如果 ORDER BY 的字段能放进联合索引，可以避免 filesort。

#### 提问：怎么找出数据库里的无用索引？

> 详细解答可以看：[MySQL 数据库的性能优化方法有哪些？](https://www.mianshiya.com/question/1816438385021997058)

回答：MySQL 5.7 以后可以用 performance\_schema 和 sys 库。`SELECT * FROM sys.schema_unused_indexes` 能列出从 MySQL 启动以来没被使用过的索引。注意这个统计会在重启后清零，最好观察一段时间再决定删除。还可以结合慢查询日志，看看哪些索引从来没出现在 EXPLAIN 的 key 列里。

#### 提问：建了很多索引，想删一些，怎么评估影响？

> 详细解答可以看：[MySQL 中使用索引一定有效吗？如何排查索引效果？](https://www.mianshiya.com/question/1780933295463231490)

回答：先用 `EXPLAIN` 把线上的核心 SQL 跑一遍，记录每条 SQL 用的是哪个索引。然后对照要删的索引，看有没有 SQL 依赖它。可以在测试环境先删掉，跑压测看性能变化。另外注意有些索引虽然查询不用，但可能是为了唯一性约束建的，删之前确认一下。
