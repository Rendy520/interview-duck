# Mysql

### MySQL 的存储引擎有哪些？它们之间有什么区别？

mysql 常用的数据引擎有 InnoDB、MyISAM、Memory、NDB、Archive

实际工作中绝大多数场景都是使用 InnoDB，InnoDB 引擎支持行级锁、事务、外键、MVCC。而且该引擎使用聚簇索引，查询速度快。

MyISAM 是早期mysql使用的引擎，只支持表级锁、不支持外键；但是因为没有事务开销（不用维护undo log）、不用支持 mvcc 的版本链查找。所以一般用在读多写少、对一致性要求不高的场景。

Memory 引擎则是专门负责内存数据的处理的，由于放在内存中有更高的读写速度，而且使用哈希索引操作更快。适用于临时表和会话缓存。

Archive 引擎凭借高压缩率专用于数据归档，只支持 insert/select，不支持索引。适用于日志、历史订单记录。

NDB 是支持分布式的集群架构专用引擎。

关键点：

1. InnoDB相对于 MyISAM 关键优化：全文索引、行级锁、MVCC、事务、聚簇索引
2. 行锁锁的是索引记录。如果有的话优先锁：二级索引（非聚簇索引），主键索引。没有索引就锁表。
3. 引擎信息可以从 information_schema.tables 里面查

### InnoDB 引擎中的聚簇索引和非聚簇索引有什么区别？

聚簇索引的B+树叶子节点存完整数据行，而非聚簇索引只存索引列和主键值。

聚簇索引一次到位。非聚簇索引如果没有查到目标需要回表（用主键值在聚簇索引查数据）

关键点：

1. 聚簇索引一般就是主键索引，可以有多个非聚簇索引
2. B+树非叶子节点只存页号、主键。叶节点内部有页目录。无需主键会页分裂导致性能变差。
3. 聚簇索引：没有主键找唯一索引，没有唯一索引建一个 6 字节隐藏列。

### MySQL 的索引类型有哪些？

从代码使用上：

- 主键索引建表时指定 primary key。
- 唯一索引、普通索引、联合索引、全文索引 可以通过 create (unique) index 创建。
- 空间索引 create spatial index。

从存储方式：

聚簇索引和非聚簇索引。

从数据结构上分：

- B+ 树索引，最常用，InnoDB 和 MyISAM 都用。
- 哈希索引，Memory、NDB 引擎用
- 全文索引，通过倒排索引技术快速定位关键词。（类似附录：词表+倒排列表）
- 空间索引，处理多维数据，支持区域查询、距离计算

关键点：

1. B+树核心优势在于范围查询（叶子之间有双链）、树矮
2. 联合索引  (a, b, c) 只有 (a) (a, b) (a, b, c) 的时候完整利用，(a, c) 时只用到a，其他都用不了
3. like %word% 如果没建索引是全表扫描（逐行找），建立全文索引是根据倒排索引查找。如果是 like word% 还能用 B+ 树索引。

### 为什么 MySQL 选择使用 B+ 树作为索引结构？

核心原因是能够减少磁盘 IO 次数达到快速存取。而能做到这一点主要是因为

1. 非叶节点不存数据。非叶节点只存主键和页号，而16 KB 的页能够支持更多的索引项，内存中能够缓存索引，提高缓存命中率，进而减少磁盘 IO
2. 叶节点双链结构。在范围索引时可以直接顺序访问链表数据，无需回根重查（相比B树），减少磁盘 IO
3. 树矮。B+ 作为多叉树，单个节点可以存 几百、几千的键值，以此推算3层 B+ 树即可存千万级数据，且每条数据最多 3 次磁盘 IO。而红黑树、avl树都是二叉树，存相同数据需要二十余层。

三者三位一体。正式因为非叶节点只存主键和页号（扩大了树的容量）、叶节点双链结构（范围查找优化）才能使树高度（平均访问次数）降低。

关键点：

1. 不用哈希是范围查找性能差（只能全表扫描）、不支持排序（order by）。InnoDB 对高频数据自动建立哈希索引进行优化访问。
2. 不用红黑树、avl 核心原因在于都是二叉树。
3. B 树也是多叉平衡树，但是节点存数据，范围查询需多次回根重查
4. 页分裂与页合并：页满会导致页分裂，主键自增之后向后追加，使用 UUID 随机天啊及会导致页分裂。
5. 主键越短非叶节点存的key越多，树越矮，IO次数越少。二级索引（非聚簇索引）同理。
6. 聚簇索引、（不同的）非聚簇索引：同文件、不同的B+树。

### MySQL 索引的最左前缀匹配原则是什么？

使用联合索引时查询条件必须从最左边开始匹配。原因是联合索引在建立时从左到右进行排序，左侧相同时根据右侧顺序排序。如果跳过最左侧字段将是无序的。

如果是多列的联合索引，只有左前缀、右后缀也可以利用左前缀，之后通过索引下推优化减少回表次数。

关键点：

1. 索引下推：以前是引擎层定位数据，返回主键给server层回表再过滤。现在直接在引擎层过滤。
2. skipscan：查右侧字段，当左前缀基数比较小的时候，可能会自动拼上最左字段查询。
3. sql 语句交换顺序没有影响
4. 联合索引：高频、区分度高、排序字段放左边。范围查询的列放右边（范围查询后有序性失效）。

### MySQL 三层 B+ 树能存多少数据？

三层大约能存两千万条记录。

InnoDB 页默认为 16 KB，假设 id 为 bigint 8B，指针 6B。非叶节点索引项数量约 1.1k。假设每条记录 1 KB。那么三层B+树的总记录数 = 1.1k * 1.1k * 16 = 2千万

关键点：

1. 设计矮，磁盘随机读大概要 10ms。16KB的页大小是权衡 IO 效率和内存利用率的结果
2. 页大：存的指针多、扇出大、树高低。内存利用率下降（数据少也是1页），更新时页都要更新（脏），写放大更严重（备份多）
3. 二级索引三层B+树存10亿多数据。8字节主键+8字节索引=1k条。1.1k * 1.1k * 1k = 10亿出头
4. page_size 如果行数据特别小可以改小，提高利用率。如果行数据大可以调大。

### MySQL 中的回表是什么？

如果使用主键索引不会产生回表。回表发生在使用二级索引时。

回表：如果查询字段没有在二级索引中全部包含，则会在二级索引所在的B+树上拿到主键后，使用主键索引查找未命中字段。

关键点：

1. 减少、避免回表：覆盖索引（查询字段都在二级索引中）、索引下推（部分命中）、减少 select *
2. 全表扫描：回表太多（数据量超过表的15%-30%）优化器放弃二级索引
3. 走索引慢：回表多
4. 回表判断：看 explain 的 extra 列。using index（覆盖索引，不回表）、using index condition（回表）。如果 type  是 index（全索引），也可能回表。

### MySQL 中使用索引一定有效吗？如何排查索引效果？

不一定有效。优化器会计算使用索引、全表扫描的 IO 和 CPU 成本进行选择，因为如果回表次数多的话使用索引可能不如直接全表扫描。

排查索引效果使用 explain 命令。在 sql 语句前加 explain，就可以看到执行计划。

1.  type 字段： all（全表扫描），range（范围扫描，扫描一段连续的区间，索引发力），ref（等值匹配，索引发力），index（全索引扫描）
2. key：索引名，null（没用索引）
3. rows：扫描行数，越大代价越高。

关键点：

1. 索引失效：联合索引不符合最左匹配、索引列运算、索引列函数、like + 左通配符、or 连接非索引字段、隐式转换、优化器判断不走、order by 不是主键也不是覆盖索引
2. 索引用上了还变慢：可以查看是不是用了错误的索引，强制使用过某个索引（force index）对比执行时间。
3. 隐式类型转换是函数转换，索引失效
4. 覆盖索引提性能：不用回表省 IO，比聚簇索引小扫描量少。

### 在 MySQL 中建索引时需要注意哪些事项？

建索引优先考虑：频繁查询的字段、排序/分组（order by，group by，distinct）字段

建索引要避免：索引太多（占空间，维护成本高）、区分度低（比如性别）、大字段（text、long text等索引空间占用大）、写多读少（频繁更新）、已被联合索引覆盖

考虑顺序：频繁查询/排序/分组，区分度，字段类型大不大，联合索引覆盖，写多读少

关键点：

1. 删索引：看使用频率（sys.schema_unused_indexes)、是否冗余（联合索引覆盖）
2. 线上加索引会锁表吗？大部分情况不会，InnoDB 会建一个链式表结构慢慢构建，只在开始结束时短暂加锁。如果数据量大还是很慢，业务低峰操作。
3. 索引合并优化：合并多个单列索引结果。但是不如联合索引。

### MySQL 中的索引数量是否越多越好？为什么？

不是。

1. 时间成本增大：
   1. 索引更新：insert，update，delete 都会触发索引更新。主键索引更新又会引发二级索引更新。
   2. 页分裂、页合并：索引更新可能会触发页分裂、页合并，写放大严重
   3. 优化器评估成本增加。
2. 空间开销增大：
   1. 每个数据页默认16KB，假设一张表1000万数据，二级索引占几百MB-几个GB。
   2. 索引增多加大内存占用，buffer pool缓存数据减少

### 如何使用 MySQL 的 EXPLAIN 语句进行查询分析？

在 select 语句前加 explain

字段：

- type 访问类型。 const（主键） > eq_ref（主键、唯一） > ref（非唯一） > range（索引范围） > index（整颗索引树） > all（全表扫描）
- Key 实际用的索引。 possible_key （候选）。 null（没有）
- rows 扫描行数。 越小好。百行、万行差距毫秒、秒
- extra。using index（覆盖索引），Using filesort （没法用索引），using temporary （临时表）

### MySQL 中如何进行 SQL 调优？

核心：减少磁盘IO。避免无效运算。

1.定位（慢查询日志）， 2.分析（explain），3.优化：

1. 索引：联合索引（少回表），最左匹配，避免索引列函数运算
2. sql：禁止 select *，避免 %like 前缀模糊查询，必定全表扫描），字符集（utf8，utf8mb4 join会隐式转换，索引无用）
3. 架构：redis（热点数据），分库分表，读写分离（从库分担压力）

关键点：

1. 索引失效：索引列函数运算、隐式转换（int-string）、or 连接非索引字段、不满足最左匹配、like 左通配、优化器判断不走过更快
2. 大表优化：分页优化、冷热数据分离

### 请详细描述 MySQL 的 B+ 树中查询数据的全过程 

B+ 树查数据看是什么查找，按走索引的等值查询：先定位到叶子节点（垂直查找），然后叶子节点内找数据行（页内查找）。

垂直查找：从根开始，比较根节点索引键值，二分确定区间，前往子节点。重复以上过程。

业内查找：默认 16 KB 数据页。InnoDB 默认用**页目录**加速查找（记录分组，每个槽指向最大记录），二分确定组然后遍历单链表找到数据。

总结：根节点 - 中间节点 - 叶子节点 - 页目录二分 - 组内链表遍历

关键点：

1. 页目录为啥不每个数据都建槽：空间成本太高（相当于多了一个数据列），每个槽2B，1k的记录就是 2KB，占总页的1/8.

### MySQL 中 count(*)、count(1) 和 count(字段名) 有什么区别？

性能上没有区别。如果 count(字段名) 时没有索引需要判断是否为 null，稍慢。

对 null 值的处理不同：count(*) 和 count(1) 统计所有行。count(字段名) 只统计该字段不为 null 的。

关键点：

1. MyISAM 为啥 count（*）贼快：只有表锁，所以有行计数器，没有条件时直接拿到。
2. InnoDB 的 count 优化：优先选择最小的索引（主键体积大，二级索引小），多线程扫描不同页
3. 大表count：redis 缓存维护计数（同步更新，处理一致性问题）、计数表、近似值（show table status），分区表并行统计

### MySQL 中 varchar 和 char 有什么区别？

长度：char 定长，varchar 变长（存多少占多少，另用1-2个字节记录长度）
性能：定长 char 更快读取
空间：varchar 省空间

关键点：
1. varchar 影响排序性能。如果varchar（n）长的话，sortbuffer放不下 需要双路排序。反之单路排序。
2. char 自动去除尾部空格
3. VARCHAR(50) 和 VARCHAR(200) 存同样的字符串，占用空间一样。但影响预分配空间
4. InnoDB char 不一定固定大小。utf8 时变长存储优化

### MySQL 是如何实现事务的？

通过四个核心组件：redo log，undo log，锁，mvcc 来保证acid

1. a原子性：undo log。每次修改前保存在undo log，回滚时要么全做，要么撤销。
2. c一致性，是aid共同作用结果，状态转移不会出现不一致。
3. i隔离性：行锁、间隙锁等锁数据，间隙锁防止幻读。mvcc保证隔离性的读写并发，读不加锁，通过undo log找目标版本，写的时候别人能读。
4. d持久性：redo log。事务提交先写在redo log，如果出现问题就重放 redo log。wal机制。 

关键点：
1. 两阶段提交：prepare（redo log），commit（binlog 写盘）

2. mvcc 能解决快照读的幻读，解决不了当前读的幻读。快照读就是普通的 select，走 MVCC 版本链，当前读比如 select ... for update，读的是最新数据，这时候要靠间隙锁来防幻读。

3. undo log 不是事务提交就能删的，要等没有事务需要用到这些旧版本了才能清理。purge 线程会扫描活跃的事务，找到最老的 ReadView，比它还老的 undo log 才能删除。所以长事务会导致 undo log 堆积，这也是为什么要避免长事务。

4. 为什么 redo log 要用循环写而不是一直追加？

   回答：redo log 的作用是崩溃恢复，checkpoint 之前的日志对应的数据页已经刷盘了，这部分日志就没用了。binlog 是一直追加的，因为它要用于主从复制和数据恢复，历史记录不能丢。

6. 幻读：行锁但是有人插入（加间隙锁）。不可重复读：删除更新数据变了（加行锁）

### MySQL 中的 MVCC 是什么？

mvcc 多版本并发控制。核心是读写互不阻塞。 写操作：不会覆盖数据，生成新版本的记录，记录保留版本号、时间戳，形成版本链。旧版本写在 undo log。 读操作：无锁的根据事务启动时间去找版本对应的 undo log。 具体实现：InnoDB 记录中有 trx_id（事务ID），roll_pointer（指向undo log）

关键点：

1. 可重复不能完全避免幻读：快照读、当前读混用。A 快照读，B 插数据，A 当前读（幻读）。
2. mvcc 不能解决写写冲突。写写冲突靠加锁。

### MySQL 中的日志类型有哪些？

三种：

1. binlog 是 server层的日志。记录逻辑操作（sql，行变前后的值）。主要用于主从同步。重放 binlog 保持数据一致。数据恢复也是。
2. redo log是 InnoDB 独有。记录物理变更后的值（某页某偏移量修改值）。保证 crash-safe，重做任务。
3. undo log 是 InnoDB的。记录数据旧值，事务回滚时用undo log改回数据。 mvcc 的快照读、事务读历史版本都找undo log。 区别：binlog记录数据变更逻辑语义，server层生成跨引擎通用。redo log，undo log 服务于 InnoDB 的事务、恢复机制。binlog 无限追加，redo log循环覆盖。

关键点：

1. redo log 写满会阻塞更新，强制推进 checkpoint，脏数据写入磁盘，之后继续写。也就是抖动现象。
